# Phase 2.1: Charts Selector and Multi-Asset Tagging - Research

**Researched:** 2026-04-13  
**Domain:** Charts selector UX + API-driven attribute composition  
**Confidence:** HIGH

<research_summary>
## Summary

Existing backend already supports:
- item resolution from asset name,
- item attributes retrieval,
- single-attribute timeseries retrieval.

Existing frontend already supports:
- page-level date filters,
- chart render/legend/fullscreen,
- chart actions and add-chart triggers.

Main gap is composition workflow: there is no modal selector, no asset search endpoint with caps, no timeseries-only attribute filtering contract, and no per-chart tag-chip removal control.

</research_summary>

<architecture_patterns>
## Recommended Patterns

- **Selector state isolation:** keep modal state local until user confirms selection.
- **Chart line contract:** represent plotted tags as normalized line descriptors (`item_id`, `attribute_id`, `label`, `asset_name`).
- **API-first filtering:** backend should enforce `timeseries_only` to avoid frontend false positives.
- **Composable retrieval:** support multi-line retrieval with deterministic ordering and per-line error tolerance.

</architecture_patterns>

<common_pitfalls>
## Common Pitfalls

1. Mixing UI-only attribute labels with backend identity fields (must keep IDs).
2. Letting non-timeseries attributes enter selection and fail at plot time.
3. Replacing existing chart interactions instead of extending them.
4. Losing selected tags when switching date presets or reopening selector.

</common_pitfalls>

<open_questions>
