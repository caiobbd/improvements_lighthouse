# Phase 2.1: Charts Selector and Multi-Asset Tagging - Context

**Gathered:** 2026-04-13  
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a Lighthouse-style selector workflow in the Charts app so users can search real assets, select timeseries-capable attributes across multiple assets, and compose chart lines through API-backed interactions. This phase also introduces per-chart attribute tag chips with explicit red `x` removal controls.

</domain>

<decisions>
## Implementation Decisions

- **D-01:** Selector opens from both `Add Chart` and per-chart `Actions`.
- **D-02:** Selector uses two-pane layout:
  - Left pane: asset list
  - Right pane: attributes of selected asset
- **D-03:** Asset search uses case-insensitive substring matching, capped at 50 results.
- **D-04:** Attribute list supports text filtering and checkbox multi-select.
- **D-05:** Only attributes with valid timeseries reference are selectable.
- **D-06:** One chart can combine selected attributes from multiple assets.
- **D-07:** Chart tags show attribute label format already used by current app naming.
- **D-08:** Each chart tag has a red `x` control to remove that series from the chart.
- **D-09:** Plot requests remain bound to the active page time filter (`7d/30d/90d/custom`).
- **D-10:** Tree hierarchy (`parent_id` / `children_id`) is deferred to a later phase.

</decisions>

<specifics>
## Specific Ideas

- Keep chart composition state explicit in chart model (selected lines with `item_id`, `attribute_id`, label).
- Keep API contract deterministic so frontend can request either:
  - per-selected line attribute timeseries, or
  - one batched endpoint for multiple selected lines.
- Preserve current legend/fullscreen interactions while adding tag-chip removal.

</specifics>
