# Phase 6.1: Alarm row parity with Lighthouse full alarm window - Context

## Decisions

- Keep this as an inserted delta phase after Phase 6, focused only on alarm row parity and narrative rendering quality.
- Do not parse prognosis or description into fixed sections because payload formats vary; render raw payload text as markdown.
- Alarm row should visually follow Lighthouse hierarchy:
  - strong alarm title
  - compact chips for key facts
  - emphasized start/elapsed/status metadata
  - narrative block below summary metadata
- Missing fields must show explicit fallback labels (`N/A`) and warning callouts instead of empty blocks.
- Keep existing alarm page generation contract from Phase 6 (window fallback logic, per-attribute charts, span overlays) unchanged.

## Discretion Areas

- Exact chip ordering and naming can be tuned for readability as long as core facts remain visible (asset, status, severity, id, start, elapsed).
- Markdown rendering strategy can use a local utility (custom lightweight renderer + sanitizer) if dependency-free integration is faster and safer for this repo.
- Collapse/expand interaction for long narrative may be default-expanded or default-collapsed; decision should prioritize fast scanability.
- Alarm row spacing may be tuned for desktop readability while preserving existing chart controls and no layout regressions.

## Deferred Ideas

- Rich references/links to related events and maintenance cases (if backend provides relational payloads later).
- Dedicated "copy alarm payload" and "open source event" quick actions.
- Optional per-card AI summary layer for long prognosis text.
- Full fullscreen alarm sheet parity with Lighthouse modal behavior.
