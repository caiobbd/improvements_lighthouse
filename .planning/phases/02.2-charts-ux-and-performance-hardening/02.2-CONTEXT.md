# Phase 2.2: Charts UX and Performance Hardening - Context

**Gathered:** 2026-04-13  
**Status:** Ready for planning

<domain>
## Phase Boundary

Harden the Charts page after first-user testing feedback. Keep Phase 2.1 feature contract intact (selector + multi-asset tags), but improve query ergonomics, long-list scrolling behavior, chart loading speed, and chart-local interaction updates.

</domain>

<decisions>
## Implementation Decisions

- **D-01:** Asset search in selector must run only on explicit user action (`Search` click or Enter submit), not on every keystroke.
- **D-02:** Selector must support reliable scrolling for large asset and attribute lists.
- **D-03:** Attribute list fetch/query must be asset-first; no eager attribute loading before asset selection.
- **D-04:** Default chart request window resolution should use `6h` sampling for faster initial load.
- **D-05:** Add temporary local client caching for repeated chart requests in same session.
- **D-06:** Tag removal should update only the affected chart, avoiding whole-page rerender experience.
- **D-07:** Chart `Actions` must open on hover and expose at least one option: `Add new tags`.
- **D-08:** Existing D3 rendering, fullscreen, and legend interactions should remain stable.

</decisions>

<specifics>
## Specific Ideas

- Use form submit in selector asset pane (`Search` button + Enter key).
- Keep asset and attribute panes with independent `overflow-y: auto`.
- Introduce request-keyed cache in API client with TTL for timeseries reads.
- Minimize rerender blast radius by targeting chart-level DOM/state updates on tag remove.
- Replace multi-option action menu with a focused menu containing `Add new tags` during this phase.

</specifics>
