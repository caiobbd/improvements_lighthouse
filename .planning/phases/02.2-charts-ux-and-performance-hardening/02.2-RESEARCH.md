# Phase 2.2: Charts UX and Performance Hardening - Research

**Researched:** 2026-04-13  
**Domain:** Selector ergonomics and chart-page performance  
**Confidence:** HIGH

<research_summary>
## Summary

Phase 2.1 delivered functional selector and multi-asset plotting, but first testing identified usability and performance bottlenecks:

- selector currently performs too many asset search calls (query-on-type),
- long list traversal needs explicit scroll reliability,
- attribute loading behavior needs strict asset-first flow,
- chart fetch path can be heavy for default period at 1h resolution,
- tag removal feels like whole-page refresh due to current render strategy,
- chart actions behavior does not match expected hover-driven menu.

This phase should focus on interaction hardening and perceived performance without changing core data contracts.

</research_summary>

<architecture_patterns>
## Recommended Patterns

- **Explicit search submit:** trigger asset query only on submit event.
- **Session cache for timeseries:** key by request params and reuse recent responses.
- **Chart-local refresh:** keep page-level state consistent while minimizing rerender scope on per-chart actions.
- **Narrow action surface:** one stable action (`Add new tags`) is preferable to several unstable entries.

</architecture_patterns>

<common_pitfalls>
## Common Pitfalls

1. Caching stale data without a clear TTL.
2. Using shared scroll containers that clip or steal wheel events.
3. Reintroducing eager attribute requests through implicit asset selection.
4. Breaking existing fullscreen/legend behaviors while optimizing chart updates.

</common_pitfalls>

<open_questions>
## Resolved by User Feedback

- Asset search trigger: explicit Search action only.
- Long list behavior: must support full scrolling for assets and tags.
- Attribute load timing: only after asset click.
- Default sampling for performance: 6 hours.
- Local performance strategy: temporary local cache is acceptable.
- Actions menu behavior: hover menu with only "Add new tags" for now.

</open_questions>
